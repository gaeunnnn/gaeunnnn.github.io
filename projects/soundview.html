가장 오래 고민한 문제와 해결 과정을 중심으로 정리했습니다.

#01

대용량 영상 업로드 시 서버 부하 발생

문제

영상 업로드 요청이 서버를 경유하면서 메모리 부하 및 업로드 지연이 발생했습니다.

원인

파일 업로드를 서버가 직접 처리하는 구조로 인해 네트워크 및 서버 자원이 과도하게 사용되었습니다.

해결

Presigned URL 기반으로 S3 직접 업로드 구조로 변경하고, Chunk 기반 병렬 업로드(Promise.all)를 적용했습니다.

결과

서버 부하를 최소화하고 업로드 속도를 49% 단축 (12.8s → 6.5s)했습니다.

#02

AI 처리 지연으로 인한 사용자 경험 저하

문제

AI 처리 시간이 길어지면서 사용자 응답이 지연되고 서비스 체감 속도가 저하되었습니다.

원인

영상 업로드와 AI 처리가 동기적으로 연결되어 요청-응답 흐름이 블로킹되는 구조였습니다.

해결

RabbitMQ 기반 비동기 메시지 큐를 도입하여 업로드와 AI 처리를 분리하고, SSE를 통해 처리 완료 시점에 사용자에게 알림을 제공했습니다.

결과

사용자는 즉시 업로드 응답을 받을 수 있게 되었으며, 서비스 응답성과 확장성이 향상되었습니다.

#03

비동기 처리 지연 시 데이터 접근 문제

문제

큐 적체로 처리 지연 시 Presigned URL 만료로 인해 AI 서버가 영상에 접근하지 못하는 문제가 발생했습니다.

원인

Presigned URL의 유효 시간 제한으로 인해 비동기 처리 시점과 접근 시점이 불일치했습니다.

해결

큐에는 URL 대신 S3 Object Key를 전달하고, 각 서버에서 도메인과 Key를 조합해 접근하도록 구조를 변경했습니다.

결과

처리 지연 상황에서도 안정적인 데이터 접근이 가능해졌으며, 대량 처리 환경에서도 높은 성공률을 유지할 수 있었습니다.

RETROSPECTIVE

배운 점

이 프로젝트를 통해 대용량 파일과 AI 처리가 결합된 서비스에서는 기능 구현보다 데이터 흐름을 어떻게 분리하고 비동기적으로 처리할지 설계하는 것이 더 중요하다는 점을 배웠습니다.

특히 업로드, AI 처리, 결과 저장처럼 여러 단계가 연결된 구조에서는 각 단계를 독립적으로 분리하고 메시지 기반으로 연결하는 것이 서비스의 안정성과 확장성에 큰 영향을 준다는 것을 체감했습니다.

또한 처리 결과를 사용자에게 전달하는 과정에서도 단순한 응답이 아닌 이벤트 기반 알림 구조를 설계하는 것이 사용자 경험을 개선하는 데 중요하다는 점을 경험했습니다.
