#01

비동기 처리 환경에서 상태 불일치 발생

문제

영상 업로드 이후 AI 처리 상태가 지연되거나 실제 처리 상태와 불일치하는 문제가 발생했습니다.

원인

업로드, AI 처리, 결과 저장이 서로 다른 시점에 수행되면서 단일 상태 기준 없이 비동기적으로 처리되어 상태 동기화가 어려웠습니다.

해결

Video 상태를 PENDING → PROCESSING → COMPLETED → FAILED로 분리하고, MQ 메시지 흐름과 DB 상태 업데이트 시점을 명확히 정의하여 상태 전이를 일관되게 관리했습니다.

결과

비동기 처리 환경에서도 상태 흐름이 명확해졌고, 사용자에게 정확한 처리 상태를 안정적으로 제공할 수 있었습니다.

#02

대용량 영상 업로드 시 서버 부하 발생

문제

영상 업로드 요청이 서버를 거치면서 트래픽 증가 시 서버 부하와 응답 지연이 발생했습니다.

원인

파일 업로드를 서버가 직접 처리하는 구조로 인해 네트워크 및 서버 자원이 과도하게 사용되었습니다.

해결

Presigned URL을 활용하여 클라이언트가 S3에 직접 업로드하도록 구조를 변경하고, 서버는 업로드 완료 이벤트만 처리하도록 역할을 분리했습니다.

결과

서버 부하가 감소하고 업로드 성능이 개선되었으며, 대용량 파일 처리에도 안정적인 구조를 확보했습니다.

#03

AI 처리 지연으로 인한 사용자 경험 저하

문제

AI 처리 시간이 길어지면서 사용자가 결과를 기다리는 동안 서비스 응답성이 떨어졌습니다.

원인

영상 처리와 AI 분석이 동기적으로 연결되어 있어 요청-응답 흐름이 블로킹되는 구조였습니다.

해결

RabbitMQ 기반 비동기 메시지 큐를 도입하여 영상 업로드와 AI 처리 과정을 분리하고, SSE를 통해 처리 완료 시점에 사용자에게 알림을 전달하도록 개선했습니다.

결과

사용자는 즉시 업로드 완료 응답을 받을 수 있게 되었고, 전체 처리 흐름이 비동기화되면서 서비스 응답성과 확장성이 향상되었습니다.

RETROSPECTIVE

배운 점

이 프로젝트를 통해 대용량 파일과 AI 처리가 결합된 서비스에서는 기능 구현보다 데이터 흐름을 어떻게 분리하고 비동기적으로 처리할지 설계하는 것이 더 중요하다는 점을 배웠습니다.

특히 업로드, AI 처리, 결과 저장처럼 여러 단계가 연결된 구조에서는 각 단계를 독립적으로 분리하고 메시지 기반으로 연결하는 것이 서비스의 안정성과 확장성에 큰 영향을 준다는 것을 체감했습니다.

또한 처리 결과를 사용자에게 전달하는 과정에서도 단순한 응답이 아닌 이벤트 기반 알림 구조를 설계하는 것이 사용자 경험을 개선하는 데 중요하다는 점을 경험했습니다.
