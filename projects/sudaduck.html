#01

동시 음성 입력 시 대화 순서 보장

문제

여러 사용자가 동시에 발화할 경우 STT 결과와 메시지 저장 순서가 뒤섞이며 실제 대화 흐름과 다른 순서로 스크립트가 구성될 가능성이 있었습니다. 단순 시간 기록만으로는 네트워크 지연이나 처리 시점 차이로 인해 순서를 안정적으로 보장하기 어려웠습니다.

원인 분석

발화 이벤트가 병렬로 들어오는 상황에서 각 요청이 독립적으로 처리되다 보니 순서를 결정할 공통 기준이 없었습니다. 특히 실시간 서비스에서는 “먼저 도착한 요청”과 “먼저 말한 사용자”가 항상 일치하지 않는 문제가 있었습니다.

해결 방식

Redis INCR 명령을 사용해 방 단위의 원자적 sequence 번호를 생성하고, 각 발화에 순서를 부여한 뒤 이를 기준으로 후속 처리 흐름을 연결했습니다. 즉, 실시간 처리 단계에서는 sequence를 기준으로 정렬 가능한 구조를 만들고, 이후 스크립트 생성이나 저장 단계에서도 같은 기준을 사용하도록 통일했습니다.

결과

동시 발화 상황에서도 실제 대화 흐름과 가까운 순서를 유지할 수 있었고, 후속 스크립트 생성 품질과 대화 리뷰의 일관성도 함께 개선할 수 있었습니다.

#02

STT 오인식으로 인한 스크립트 품질 저하

문제

사용자의 실제 대화가 STT 과정에서 잘못 인식되면, 이후 생성되는 영어 스크립트와 복습 문장도 함께 왜곡되는 문제가 있었습니다. 이는 단순한 오타 수준이 아니라, 학습 서비스 전체 품질을 떨어뜨릴 수 있는 문제였습니다.

원인 분석

음성 인식 결과에는 잡음, 발화 중복, 불완전한 문장, 맥락과 맞지 않는 오인식 단어가 포함될 수 있었고, 이 상태로 곧바로 LLM에 전달하면 잘못된 입력이 그대로 확장되는 구조였습니다.

해결 방식

규칙 기반 필터링과 후처리 로직을 먼저 적용해 불필요한 토큰, 반복 표현, 비정상 문장을 제거하는 전처리 파이프라인을 구성했습니다. 이후 정제된 문장만 영어 스크립트 생성 단계로 넘기도록 분리하여 “인식 결과”와 “학습용 텍스트” 사이에 품질 보정 단계를 두었습니다.

결과

영어 스크립트 생성 정확도가 개선되었고, 사용자 입장에서도 대화 결과가 더 자연스럽고 학습 가능한 문장으로 제공되는 효과를 얻었습니다.

#03

실시간 상태와 영속 데이터의 책임 분리

문제

방 준비 상태, 참여자 상태, 실시간 turn 정보처럼 즉시 반응이 필요한 데이터와 학습 기록, 저장된 문장처럼 최종적으로 보존되어야 하는 데이터를 같은 저장소와 흐름으로 처리하면 구조가 복잡해지고 응답 속도에도 부담이 생길 수 있었습니다.

원인 분석

모든 데이터를 DB 중심으로 처리하면 즉시성이 떨어지고, 반대로 모든 데이터를 Redis에 두면 영속성과 관리 책임이 모호해집니다. 결국 “어떤 데이터가 즉시성 중심인지, 어떤 데이터가 보존 중심인지”를 먼저 구분하는 설계가 필요했습니다.

해결 방식

실시간 반응이 필요한 상태 데이터는 Redis 중심으로 관리하고, 최종 보존이 필요한 데이터만 DB에 저장하는 방향으로 책임을 분리했습니다. 이를 통해 실시간 이벤트 처리 속도를 확보하면서도, 학습 결과와 문장 데이터는 안정적으로 영속화할 수 있는 구조를 만들었습니다.

결과

상태 조회와 이벤트 반영 속도가 개선되었고, 데이터 저장 구조도 훨씬 명확해져 유지보수성과 확장 가능성을 함께 확보할 수 있었습니다.

RETROSPECTIVE

배운 점

이 프로젝트를 통해 실시간 서비스에서는 기능 구현 자체보다 상태를 어떻게 정의하고, 어떤 저장소에 두며, 이벤트를 어떤 기준으로 전파할지를 먼저 설계하는 일이 훨씬 중요하다는 점을 배웠습니다. 또한 음성 처리와 학습 생성처럼 여러 단계가 연결되는 서비스에서는 파이프라인을 명확히 나누고 중간 정제 단계를 설계하는 것이 최종 품질에 직접적인 영향을 준다는 점을 체감했습니다.
