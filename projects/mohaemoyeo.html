#01

Plan 중심 관계형 데이터 구조 설계

문제

초기 설계에서는 일정 데이터를 사용자 중심으로 관리해 여러 사용자가 하나의 계획에 참여하는 구조를 표현하기 어려웠습니다.

원인

users 와 plans 관계만으로는 작성자와 참여자를 구분하기 어려워 공동 일정 관리 로직이 복잡해졌습니다.

해결

plans 를 중심 엔티티로 설정하고 참여 사용자 관리를 plan_participants 테이블로 분리했습니다. 또한 role 컬럼으로 참여자 권한을 구분하도록 설계했습니다.

결과

플랜 생성자와 참여자 역할이 명확해졌고 공동 일정 조회·참여·수정 흐름을 일관된 데이터 구조로 처리할 수 있었습니다.

#02

이미지 데이터 분리 설계로 확장성 개선

문제

플랜과 댓글에 여러 장의 이미지가 첨부되면서 이미지 데이터를 본 테이블에 함께 저장하면 조회 구조가 복잡해졌습니다.

원인

이미지를 plans 또는 comments에 직접 포함하면 다중 이미지 관리와 정렬 처리가 어려워졌습니다.

해결

이미지 데이터를 plan_photos, comment_photos 테이블로 분리하고 원본 엔티티와 1:N 관계 구조로 설계했습니다. 또한 order_no 컬럼으로 이미지 순서를 관리했습니다.

결과

본문과 이미지 데이터를 분리해 목록 조회 성능을 개선하고 이미지 확장과 정렬 처리를 유연하게 대응할 수 있었습니다.

RETROSPECTIVE

배운 점

소셜 기반 서비스에서는 단순한 기능 구현보다 도메인 관계를 데이터 구조로 어떻게 모델링할 것인지가 중요하다는 점을 배웠습니다.
특히 하나의 plan을 중심으로 사용자, 참여자, 댓글, 이미지 등 여러 엔티티가 연결되는 구조를 설계하면서 모든 데이터를 하나의 테이블에 포함하기보다 역할에 따라 엔티티를 분리하고 관계를 명확히 정의하는 것이 API 설계, 데이터 조회 구조, 유지보수성 측면에서 더 효율적이라는 것을 경험할 수 있었습니다.
