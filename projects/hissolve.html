TROUBLE SHOOTING

문제 해결 과정

기능 구현보다 시스템 신뢰성에 더 직접적으로 연결된 이슈들을 중심으로 정리했습니다.

#01

Connection Pool 계산 오류와 deadlock 발생

문제

HikariCP의 maxPoolSize를 150으로 설정했지만 실제 테스트 과정에서 deadlock이 발생했습니다. 당시에는 client 하나당 connection 하나만 사용할 것이라 가정했지만, 실제 요청 처리 흐름은 그보다 단순하지 않았습니다.

원인 분석

connection 사용량을 단순히 “사용자 수 = connection 수”로 계산하면 안 된다는 점을 확인했습니다. 요청 처리 과정, 트랜잭션 범위, 동시에 실행되는 로직이 겹치면서 예상보다 더 많은 connection이 점유될 수 있었고, 이것이 deadlock과 waiting 상황으로 이어졌습니다.

검증 방식

단순히 pool 크기를 키우는 대신, 오히려 Hikari max pool size를 1로 낮춰 테스트를 진행했습니다. 이를 통해 어느 지점에서 deadlock이 발생하는지, connection timeout waiting이 얼마나 발생하는지를 더 명확하게 관찰할 수 있었습니다.

결과

connection pool은 숫자만 크게 잡는 문제가 아니라, 요청 구조와 트랜잭션 설계를 함께 보고 검증해야 한다는 점을 배웠습니다. 이 경험을 통해 운영 환경 기준으로 자원 사용을 분석하는 시각을 갖게 되었습니다.

#02

서로 제출한 코드가 엉키는 현상

문제

여러 사용자가 동시에 코드를 제출할 경우, 컴파일을 위해 생성되는 임시 파일이 같은 디렉토리를 공유하면서 다른 사용자의 코드와 실행 결과가 섞이는 문제가 발생했습니다.

기존 방식

모든 사용자가 동일한 임시 경로를 바라보는 구조였습니다. compile 시간이 겹치면 같은 위치의 파일을 덮어쓰게 되어 충돌이 발생할 수 있었습니다. 

String sharedDir = "/tmp/tempDir";

해결 방식

제출마다 128비트 기반 UUID를 붙여 독립된 임시 디렉토리를 생성하도록 수정했습니다. 이를 통해 각 사용자의 제출이 완전히 분리된 공간에서 컴파일되도록 바꿨습니다. 

String uniqueDirPath = "/tmp/tempDir/" + UUID.randomUUID();

결과

동시에 제출이 들어와도 파일 충돌 없이 각 제출을 독립적으로 처리할 수 있게 되었고, 실행 환경의 안정성과 제출 결과의 신뢰성을 높일 수 있었습니다.

RETROSPECTIVE

배운 점

Hissolve를 통해 기능이 동작하는 것만으로는 충분하지 않고, 동시에 많은 요청이 몰리는 상황에서도 얼마나 안정적으로 동작하는지가 플랫폼의 신뢰성을 결정한다는 점을 배웠습니다. 특히 제출 처리처럼 자원 경쟁이 발생할 수 있는 서비스에서는 실행 환경 분리, connection pool 검증, 병목 구간 분석 같은 운영 관점의 문제 해결이 백엔드 개발자의 중요한 역할이라는 점을 체감했습니다.
